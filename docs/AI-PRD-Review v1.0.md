## 潛在風險

### 相互衝突的需求

1. **部署方式（GitHub Pages）vs. 系統功能需求（多角色登入、資料持久化、審核流程狀態）——最嚴重的衝突**
   GitHub Pages 僅提供靜態檔案託管，無法執行後端邏輯或提供資料庫。但系統要求：登入驗證、跨使用者（員工/主管/HR）共享同一份請假資料、審核狀態變更需被持久保存。若技術棧不新增後端或 BaaS（如 Firebase/Supabase），這些核心功能在現行部署策略下**無法真實實現**，只能做成 mock data／LocalStorage 的展示版本。這個決策會直接影響 MVP 範圍與所有功能需求的設計方式，建議優先釐清。

2. **業務目標「降低 HR 人工維護成本」vs. 未定義的資料來源**
   若資料僅存在瀏覽器端（mock/LocalStorage），並不會真正減少 HR 的人工作業，與商業目標矛盾，除非明確定位為 Demo/Portfolio 專案。

3. **專案定位模糊：真實企業系統 vs. 技術展示作品集**
   專案描述同時強調「解決公司真實請假流程問題」與「展示 Vue3 + AI-Driven Development」。這兩種定位對範圍、安全性、合規（勞基法試算）的要求完全不同，PRD 需要明確二選一或分階段（先做 Demo、後做正式版）。

4. **主管目標「了解團隊休假安排」vs. 缺乏對應功能需求**
   目標存在但無對應使用者故事，未來若補上團隊行事曆功能，可能與目前僅聚焦「個人申請 + 逐筆審核」的 MVP 範圍產生排程/範圍衝突。

5. **優先權缺乏區分度**：US-001～003 全部標示「高」，沒有 P0/P1/P2 分層，一旦 US-004～008 補齊後仍全部標「高」，將無法有效指導 MVP 的取捨。

### 技術風險

1. **部署/後端不匹配**（同上，影響最大）。
2. **身分驗證安全性**：若無後端，登入僅能是前端模擬驗證，無法安全驗證帳密，若混用真實員工資料將有資安疑慮。
3. **資料持久化策略未定義**：Pinia 僅管理記憶體內狀態，重新整理頁面即遺失資料；多使用者同時存取（HR 與主管看同一批資料）在純前端架構下無法實現。
4. **API 需求章節空白**：前端無法針對明確的 API 合約開發 service 層，未來串接後端時整合風險高。
5. **請假規則的業務邏輯複雜度**：若需真實反映台灣勞基法的特休/病假規則，這是一套獨立的規則引擎，若不及早在架構上抽離，容易散落在各元件中難以維護。
6. **RBAC（角色權限）設計未定義**：三種角色的畫面/操作權限如何控管（路由層級 + UI 層級）未有說明，風險是權限判斷邏輯散落各處。
7. **UI/UX 需求空白**：無法確認是否有無障礙（a11y）、Element Plus 客製化程度、響應式斷點等要求，實作階段容易產生不一致的設計決策。
8. **非功能需求空白**：效能、瀏覽器支援範圍、可用性等指標缺失，難以驗證技術決策是否符合預期。
9. **缺乏測試策略**：搭配「AI-Driven Development」的敘事，若無單元測試/E2E 測試作為安全網，AI 輔助重構的風險會提高。
10. **日期/時區處理與工作日計算**：請假天數是否排除週末與國定假日，若未及早規劃，日期計算（尤其半天假）容易產生大量邊界案例錯誤。
11. **AI 工作流程（Claude Code + Notion MCP + GitHub MCP）目前只是口號**：PRD 沒有具體說明這些工具如何嵌入實際交付流程（例如 Issue 自動建立、PR 自動審查），若不落地成具體流程，這項展示賣點可能無法被驗證。

## 改進建議

### PRD 文件面

1. **優先補齊六個空白章節**（功能需求、非功能需求、UI/UX 需求、API 需求、驗收標準、里程碑），這是目前最大的落地障礙。
2. **為 US-004～008 補上完整五要素**（故事內容、驗收標準、優先權、依賴項），格式比照 US-001～003。
3. **導入優先權分級（P0/P1/P2 或 MoSCoW）**，取代目前全部「高」的扁平標示，明確界定 MVP 的邊界。
4. **明確決策部署與後端策略**：若維持 GitHub Pages，建議明確定位為「前端展示 + Mock Data/LocalStorage」的 Demo 專案；若要做成可真實使用的系統，需在技術棧中新增後端/BaaS（如 Firebase、Supabase 或自建 API），並更新非功能需求與 API 需求章節。這個決定應該最優先拍板，因為會牽動幾乎所有其他需求的設計。
5. **補上假別規則、組織架構、通知機制、統計報表**四大業務需求區塊的具體規格，這些是請假系統的核心，目前完全缺席。
6. **釐清專案定位**（真實系統 vs. 作品集展示），並在文件開頭明確聲明，避免範圍認知落差。

### MVP 範圍評估

目前隱含的 MVP（US-001～008 涵蓋登入、查看餘額、申請、查看紀錄、主管審核、HR 管理假別）**方向大致合理**，涵蓋三種角色的核心互動，作為第一階段範圍是恰當的。但在鎖定 MVP 前，建議務必：

- **先決定後端/資料持久化策略**（見上），否則無法估算真實工時，目前的 MVP 可能只是「UI 原型」而非「可運作系統」。
- **將組織架構（員工-主管對應關係）納入 MVP 基礎建設**，否則審核流程無法路由到正確的主管，這是容易被低估、但屬於基礎設施等級的必要功能。
- **明確排除**（Out of Scope）團隊行事曆、統計報表、多層審核、通知機制等項目，或是評估是否至少要有「陽春版」通知（即使只是站內訊息），因為這是現況五大痛點之一，若 MVP 完全不處理，展示效果會打折。
- 在未補齊 FR/NFR 前，目前還無法給出「MVP 範圍合理」的最終確認，建議文件補齊後再做一次範圍複核。

### 使用者故事 → GitHub Issues 轉換檢查

- **US-001、US-002、US-003：可直接轉換**。已具備角色、故事、驗收標準、優先權、依賴項，符合一般 Issue Template 所需的最低要素。轉換時建議額外補上：對應的 FR 章節連結、Label（角色/優先權）、Definition of Done、以及是否需要 Mock API 或真實 API。
- **US-004～008：目前無法轉換**。只有標題與角色，工程師無從得知範圍與完成定義，必須先補齊五要素才能建立可執行的 Issue，否則會產生範圍模糊的「幽靈 Issue」。
- 建議在轉換流程中統一使用「一個使用者故事 = 一個 Epic，拆解為多個技術 Task（UI 元件、Store、Service、路由守衛等）」的兩層結構，方便追蹤與 AI 輔助開發（Claude Code）逐一認領。

## 推薦的模組結構

考量 Vue3 + Vite + TypeScript + Pinia + Vue Router + Element Plus，且系統本質是「三種角色、RBAC 導向」的應用，建議採用「以業務領域為主、型別為輔」的模組劃分（Feature-based / Vertical Slice），而非單純依檔案型別分資料夾：

```
src/
├── app/                      # 應用啟動：main.ts、App.vue、全域 Provider
├── router/                   # 路由定義、角色權限守衛（route guard）、依角色分群的路由設定
├── stores/                   # Pinia，依業務領域劃分（非依頁面）
│   ├── auth.store.ts
│   ├── leaveBalance.store.ts
│   ├── leaveRequest.store.ts
│   ├── approval.store.ts
│   ├── leaveTypes.store.ts
│   └── user.store.ts
├── services/                 # API 存取層（可替換：mock ↔ 真實後端），依領域拆分
│   ├── authService.ts
│   ├── leaveService.ts
│   ├── approvalService.ts
│   └── reportService.ts
├── features/                 # 依角色/業務能力垂直切分的功能模組
│   ├── auth/                 # 登入頁、auth composables
│   ├── employee/
│   │   ├── leave-balance/
│   │   ├── leave-request/    # 申請/編輯/取消請假表單
│   │   └── leave-history/
│   ├── manager/
│   │   ├── approval-queue/
│   │   └── team-calendar/
│   └── hr/
│       ├── leave-types-admin/
│       ├── employee-management/
│       └── reports/
├── components/                # 共用、無業務邏輯的 UI 元件（Element Plus 的封裝層）
├── composables/                # 可複用邏輯（usePermission、useDateRange、usePagination 等）
├── layouts/                    # 依角色區分的版面外殼（或單一版面 + 導覽設定）
├── directives/                 # 如 v-permission，統一處理 RBAC 顯示邏輯
├── types/                      # 共用型別/列舉：LeaveType、LeaveStatus、Role、User
├── utils/                      # 日期/工作日計算、格式化、驗證邏輯
├── constants/                  # 狀態列舉、角色列舉、路由名稱常數
├── mocks/                      # 若後端未就緒：Mock API / 種子資料，供 Demo 模式使用
└── locales/                    # 若需多語系
```

**架構重點說明：**

1. **以角色/業務領域為主要切分軸**（`features/employee`、`features/manager`、`features/hr`），而非把所有元件塞進單一 `components/`、`views/`——三種角色的功能會持續分化，垂直切分更利於後續擴充與程式碼歸屬清晰。
2. **Service 層需與 UI 解耦**，讓「Mock Data → 真實後端」的切換（見上方部署風險）只需替換 `services/` 內部實作，不需改動任何元件。這是應對目前部署策略未定的關鍵緩衝設計。
3. **RBAC 集中管理**：透過路由 meta + 導覽守衛 + `usePermission`/`v-permission`，避免權限判斷邏輯（`v-if="role === 'manager'"`）散落在各元件中。
4. **Pinia store 依業務邊界拆分**，避免單一「請假 Store」同時承擔餘額、申請、審核三種職責而過度肥大。
5. **日期/工作日計算獨立成 `utils`**，因為員工申請表單、主管審核畫面、HR 報表都會共用同一套「排除假日計算天數」邏輯，必須單一事實來源，避免重複實作導致計算不一致。
